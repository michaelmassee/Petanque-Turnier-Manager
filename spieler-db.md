# Spieler-Datenbank mit Erfassungsmaske für Meldeliste

> **Status (aktuelle Session): vollständig implementiert.**
> Alle 3 Schritte sind im Branch `spieler-db` umgesetzt. 21/21 Repository-Tests grün, SpotBugs für `spielerdb/` sauber, OXT-Plugin baut. Bewusste Abweichungen vom Originalplan sind unter „Architektur-Abweichungen" dokumentiert.

## Architektur-Abweichungen (im Code so umgesetzt)

1. **Reines JDBC-HSQLDB statt `.odb`-via-`XDatabaseContext`.** Die DB liegt unter `~/.petanqueturniermanager/spielerdb/` als HSQLDB-File-Modus (`db.script`/`db.properties`/`db.lck`). Vorteil: Repos sind ohne LO testbar, kein Lock-Konflikt mit Base, kein UNO-DataSource-Bootstrap. Trade-off: User kann die DB nicht direkt in LO Base öffnen.
2. **Connection-Lifecycle als Singleton + JVM-Shutdown-Hook** statt Eingriff in `PetanqueTurnierManagerImpl.dispose()` (existiert nicht). HSQLDB schreibt beim Shutdown ein `SHUTDOWN`-Statement und persistiert sauber.
3. **Vereinsspalte: Header-Scan statt `IMeldeliste`-Interface-Erweiterung.** Der `SheetMeldelisteAdapter` scannt die Header-Zeile auf bekannte Hinweise (`verein`, `vereinsname`, `club`, `vereniging`). Damit entfällt der ursprünglich geplante Eingriff in 5 System-Klassen — robuster gegen Refactorings, ein zentraler Ort.
4. **Tests: JUnit Jupiter 5** (CLAUDE.md-Notiz „JUnit 4" war veraltet; das Projekt nutzt bereits Jupiter).
5. **HSQLDB-Dependency:** `org.hsqldb:hsqldb:2.7.4` als `implementation`. Zusätzlich `exclude: ['hsqldb.jar', 'sdbc_hsqldb.jar']` für die LO-Class-Path-Pakete im build.gradle — `sdbc_hsqldb.jar` zog via Manifest-`Class-Path` die uralte `/usr/share/java/hsqldb1.8.0.jar` rein.

## Context

Heute wird die Meldeliste in jedem Turniersystem manuell mit Nr+Name befüllt. Es gibt keine Wiederverwendung von Spielerdaten zwischen Turnieren — jeder Spieler wird bei jedem neuen Turnier neu eingetippt. Vereins- und Lizenzinformationen existieren überhaupt nicht.

Ziel: Eine zentrale Spieler-DB als Stammdatenpool, plus Erfassungsmaske, die per Suche Spieler aus der DB in die aktuelle Meldeliste übernimmt. Das Meldeliste-Sheet bleibt strukturell unverändert (nur Nr + Spielername) — kein Eingriff in die 6 bestehenden Turniersysteme.

## Entscheidungen (mit User abgestimmt)

- **DB-Quelle:** Embedded HSQLDB. Plugin legt eine `petanque-spieler.odb` im Plugin-User-Verzeichnis an, falls nicht vorhanden (One-Click).
- **Verein als eigene Tabelle** mit nur `NR + NAME UNIQUE` (FK von SPIELER, `ON DELETE RESTRICT`).
- **Spielerfelder:** `nr` (PK, autoincrement), `vorname`, `nachname`, `verein_nr` (FK, nullable), `lizenznr`.
- **Maske:** Such-/Übernahme-Dialog mit Trefferliste; Button „Neuer Spieler" öffnet Erfassungsformular; Doppelklick übernimmt Treffer in nächste freie Meldeliste-Zeile.
- **Meldeliste-Layout:** unverändert. Übernahme schreibt `"Vorname Nachname"` in die bestehende Name-Spalte. Vereinsname wird in die optionale Vereins-Spalte mitgeschrieben, **falls** sie für das aktuelle Turniersystem aktiviert ist (existiert bereits in Schweizer/KO/JGJ/Maastrichter/Poule).

### Datenbank-Bootstrap

- Pfad: `<userconfig>/Scripts/python/petanque/petanque-spieler.odb` (analog Sidebar-Userdata-Pattern).
- Beim ersten Aufruf: `XDatabaseContext` (`com.sun.star.sdb.DatabaseContext`) → `createInstance()` → HSQLDB-URL setzen → `storeAsURL()`.
- Schema (idempotent via `CREATE TABLE IF NOT EXISTS`):
  ```sql
  CREATE TABLE IF NOT EXISTS VEREIN (
    NR INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    NAME VARCHAR(150) NOT NULL UNIQUE
  );

  CREATE TABLE IF NOT EXISTS SPIELER (
    NR INTEGER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    VORNAME VARCHAR(100) NOT NULL,
    NACHNAME VARCHAR(100) NOT NULL,
    VEREIN_NR INTEGER,
    LIZENZNR VARCHAR(50),
    CONSTRAINT FK_SPIELER_VEREIN FOREIGN KEY (VEREIN_NR)
      REFERENCES VEREIN(NR) ON DELETE RESTRICT
  );
  CREATE INDEX IF NOT EXISTS IDX_SPIELER_NACHNAME ON SPIELER(NACHNAME);
  CREATE INDEX IF NOT EXISTS IDX_SPIELER_VEREIN   ON SPIELER(VEREIN_NR);
  -- Eindeutigkeit der Lizenznr (NULL erlaubt, Mehrfach-NULL erlaubt) als letzte Instanz gegen Race Conditions
  CREATE UNIQUE INDEX IF NOT EXISTS IDX_SPIELER_LIZENZ
    ON SPIELER(LIZENZNR) WHERE LIZENZNR IS NOT NULL;
  ```
- Verbindung über `XDataSource.getConnection("","")`. Lifecycle: lazy, beim Plugin-Stop schließen (in `PetanqueTurnierManagerImpl.dispose()`).

### UI-Dialoge (UNO AWT)

Alle Dialoge erben von `AbstractUnoDialog` (`konfigdialog/`-Pattern, vgl. `FarbenDialog`).

**`SpielerSucheDialog`:**
- Eingabefeld `txtSuche` (Name/Lizenz, Live-Filter on `XTextListener.textChanged`, Trigger ab 2 Zeichen).
- **Suche als Prefix-Match**, indexfreundlich: `WHERE LOWER(NACHNAME) LIKE LOWER(? || '%') OR LOWER(VORNAME) LIKE LOWER(? || '%')`. Kein `%term%` (Full-Scan ab ~5k Spielern spürbar).
- `XListBox` oder `XGrid` mit Spalten: Nachname, Vorname, Verein, Lizenz (aus `findByNamePart`-JOIN).
- Buttons: „In Meldeliste übernehmen", „Neuer Spieler …", „Bearbeiten …", „Löschen", „Schließen".
- **Löschen IMMER mit Bestätigungsdialog** (`MessageBox` Yes/No, Default = No): „Spieler 'Vorname Nachname' wirklich löschen?". Erst nach Bestätigung `SpielerRepository.delete(nr)`. Kein Soft-Delete in v1 — Undo gibt es nicht, deshalb harter Confirm.

**Dialog-Modi & Aktive-Meldeliste-Erkennung:**
- Dialog-Aufruf via Menü `spielerdb_oeffnen` → **Verwaltungsmodus**: Übernehmen-Button ausgeblendet/disabled, Titel „Spieler-Verwaltung". Nur CRUD.
- Dialog-Aufruf via Menü `spielerdb_in_meldeliste` → **Übernahme-Modus**: Erkennung der aktiven Meldeliste über `WorkingSpreadsheet` + Sheet-Name-Detection (analog `BlattschutzRegistry`-Lookup). Wenn keine Meldeliste aktiv: MessageBox „Bitte zuerst eine Meldeliste eines Turniersystems öffnen." → Fallback auf Verwaltungsmodus oder Dialog-Abbruch (i18n-Key `spielerdb.fehler.keine_meldeliste`).
- Im Übernahme-Modus zeigt die Titelzeile zusätzlich Turniersystem + Formation an („Supermelee — Mêlée" / „Schweizer — Triplette").

**Formationsbewusste Block-Übernahme (TETE/DOUBLETTE/TRIPLETTE):**
- Dialog liest `IMeldeliste.getFormation()` der aktiven Meldeliste.
- TETE / MELEE: Doppelklick übernimmt 1 Spieler in nächste freie Zeile. Sofortige Aktion.
- DOUBLETTE: Modus „Team eintragen (2 Spieler)". Übernehmen-Button öffnet kurzes Sammel-Panel (kleine Liste der bisher gewählten Team-Spieler) — Doppelklick fügt zur Auswahl hinzu, ab 2 Spielern wird Button „Team in Meldeliste schreiben" aktiv → schreibt beide Zeilen als Block. „Verwerfen" leert die Auswahl.
- TRIPLETTE: analog mit 3 Spielern.
- Vorteil: Triplette-Partner werden garantiert in aufeinanderfolgende Zeilen geschrieben (Voraussetzung für die Teambildungs-Logik der Algorithmen).
- Block-Schreiben weiterhin Pflicht über `RangeHelper`/`RangeData`/`RowData` — auch bei einzelnen Zeilen, kein zellenweises Schreiben (CLAUDE.md).

**Doppelten-Schutz:**
- Vor jeder Übernahme: `MeldeListeHelper` liest aktuelle Spielernamen aus der Meldeliste, Vergleich case-insensitiv getrimmt gegen `"Vorname Nachname"`.
- Match → MessageBox Yes/No (Default No): „Spieler 'X' steht bereits in Zeile N. Trotzdem hinzufügen?" (i18n-Key `spielerdb.frage.spieler_doppelt`).
- Bei Block-Modus (Doublette/Triplette): Check beim Hinzufügen zur Sammel-Auswahl, nicht erst beim finalen Schreiben.

**Vereinsname mit übernehmen:**
- Wenn das Turniersystem die Vereinsname-Spalte aktiviert hat (existierende Konfig-Property pro System), wird beim Übernehmen der Vereinsname in dieselbe Zeile geschrieben.
- Erkennung: neue Helper-Methode `IMeldeliste.getVereinsnameSpalte()` — liefert `Optional<Integer>` (Spaltenindex, leer falls deaktiviert). Default-Implementation: leer; betroffene Systeme (Schweizer/KO/JGJ/Maastrichter/Poule) überschreiben und lesen ihre vorhandene Konfig-Property.
- Spieler ohne zugeordneten Verein → leere Zelle in Vereinsspalte (kein Default).
- `RowData` enthält dann beide Werte (Name + Verein) als ein Block-Schreibvorgang.
- Übernahme: nächste freie Zeile in Meldeliste suchen (analog `MeldeListeHelper`), via `RangeHelper`/`RangeData` als Block schreiben — **nicht** zellenweise (CLAUDE.md).

**`SpielerErfassenDialog`:**
- Textfelder: Vorname, Nachname, Lizenznr.
- Verein: `XComboBox` mit Autocomplete aus `VereinRepository.findAll()`. Freitext erlaubt — beim Speichern wird der Eingabewert **getrimmt** (`String.strip()`, interne Doppel-Leerzeichen kollabieren), dann case-insensitiv per `findByName` abgeglichen (existierender Verein wiederverwenden, kein Duplikat trotz UNIQUE). Existiert er nicht: Rückfrage „Verein 'X' anlegen?" → `VereinRepository.insert(getrimmt)` → neue `VEREIN_NR` setzen. Leer lassen erlaubt (`VEREIN_NR = NULL`). Fuzzy-Matching (Levenshtein o.ä.) ist explizit Out-of-Scope für v1.
- OK/Abbrechen.
- Validierung: Vorname+Nachname Pflicht; Lizenz optional, aber wenn gesetzt → unique-Check.

**`VereinVerwaltenDialog`:**
- Liste aller Vereine.
- Anlegen / Bearbeiten / Löschen.
- Beim Löschen: zuerst per `SpielerRepository.countByVerein(vereinNr)` prüfen — wenn >0, Fehlermeldung mit Anzahl betroffener Spieler (RESTRICT-Hinweis) statt Stacktrace.
- **Wenn 0 Spieler zugeordnet:** Bestätigungsdialog Yes/No (Default = No) „Verein 'Name' wirklich löschen?", erst dann `delete`.

## Risiken / Open Points

1. **HSQLDB-Lock-File:** Embedded HSQLDB hält einen exklusiven Lock auf die DB-Datei (`.lck`). Öffnet der User die `.odb` parallel in LO Base, schlägt unsere Connection fehl. Mitigation in `SpielerDbConnection`:
   - `SQLException` fangen und auf typische HSQL-Lock-Marker prüfen (Message enthält `"locked"` / `"database is already in use"` bzw. SQLState `S1000`/`08001`).
   - Im Lock-Fall lokalisierte Fehlermeldung via `MessageBox` mit klarer Handlung: „Die Spieler-Datenbank ist bereits in LibreOffice Base geöffnet. Bitte schließen und erneut versuchen." (i18n-Key `spielerdb.fehler.db_locked`).
   - Andere `SQLException` weiterhin als generischer DB-Init-Fehler (`spielerdb.fehler.dbinit`) mit Stacktrace im Log (CLAUDE.md: keine swallowed Exceptions).
   - Retry/Timeout-Schleife bewusst nicht — der Lock geht nur durch User-Aktion (Base schließen) weg, Polling bringt nichts.
2. **Format des übernommenen Namens:** Default `"Vorname Nachname"`. Bei Bedarf später konfigurierbar.
3. **Nr-Konflikt:** Spieler-DB-`NR` ≠ Meldeliste-`Spieler-Nr`. DB-Nr ist nur interner PK, Meldeliste vergibt eigene laufende Nr.
4. **NullAway:** Neues Paket sofort `@NullMarked`. JDBC-Returns sauber als `@Nullable`. `SpielerDatensatz.vereinNr` ist `@Nullable Integer`.
5. **SpotBugs:** Connection-/PreparedStatement-/ResultSet-Handling MUSS try-with-resources, sonst `OBL_UNSATISFIED_OBLIGATION`.
6. **RESTRICT-UX:** Beim Vereins-Löschen erst `countByVerein` prüfen, nicht auf SQL-Exception verlassen — gibt klarere Fehlermeldung.
